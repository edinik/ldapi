# 增加数据备份功能

## Goal

为 LDAPI 提供管理员可操作、可验证的数据备份下载能力，降低 SQLite 文件损坏、误操作或部署迁移造成的数据丢失风险。

## Background

- 项目使用 SQLite / `better-sqlite3`，生产数据库固定为 `data/sqlite.db`，运行时启用 WAL。
- Docker Compose 将宿主机 `./data` 挂载到容器 `/app/data`；当前 README 只提示运维人员手工备份目录。
- 现有“模型导入”仅覆盖模型实体，不是完整数据库备份。
- 数据库包含管理员密码哈希、TOTP 密钥、会话、AI API Key、站点、模型、资源及关联数据，备份文件属于敏感数据。

## Requirements

### R1. 备份范围与一致性

- 备份覆盖当前 SQLite 数据库的完整 schema 和持久业务数据。
- 必须通过 SQLite 在线备份机制生成一致性快照，不能在 WAL 写入期间直接复制活动数据库文件。
- 备份副本保留 `sessions` 表结构但清空会话记录；源数据库和当前登录会话不得被修改。
- 返回下载前必须对备份执行 SQLite 完整性检查。

### R2. 管理入口

- 新增独立的 `/admin/backup`“数据备份”页面，并从 `/admin` 后台首页提供入口。
- 页面说明备份覆盖范围、敏感数据风险、安全存放要求以及恢复后需要重新登录。
- 管理员通过页面手动触发备份；MVP 不提供定时备份、NAS 上传或服务端备份历史。

### R3. 认证与敏感操作保护

- 页面和下载接口都必须要求有效管理员会话。
- 下载前必须重新验证当前管理员密码。
- 当前管理员启用 TOTP 时，还必须验证有效 TOTP；未启用时不要求验证码。
- 重新认证不重复执行 Turnstile。
- 认证凭据、session ID、TOTP secret 和备份内容不得写入日志。

### R4. 下载与文件处理

- 下载产物为未加密的标准 `.sqlite` 文件，以保持恢复工具兼容性。
- 文件名使用 `ldapi-backup-<UTC timestamp>.sqlite` 格式，并避免同次下载冲突。
- 下载响应必须禁止缓存，并使用附件响应头。
- 生产环境只允许通过 HTTPS 生成备份；开发环境可使用 HTTP。
- 允许使用唯一的短生命周期临时文件，但无论成功或失败都必须清理，应用服务器不得保留备份历史。

### R5. 恢复与文档

- MVP 不提供应用内上传、覆盖或合并数据库的恢复功能。
- 文档说明停止应用、保留当前数据库副本、替换 `data/sqlite.db`、重新启动、验证和回滚的恢复流程。
- 恢复文档明确说明备份中的 session 已清空，恢复后需要重新登录。

## Acceptance Criteria

- [x] AC1: 已登录管理员可从后台首页进入 `/admin/backup`；未登录访问页面会跳转到登录页。
- [x] AC2: 下载接口对缺失或无效 session 返回既有 API 认证错误，不生成备份。
- [x] AC3: 密码错误时拒绝下载；启用 TOTP 时，缺少或错误验证码也拒绝下载。
- [x] AC4: 正确重新认证后下载标准 SQLite 文件，响应包含附件文件名和禁止缓存头。
- [x] AC5: 下载文件通过 `PRAGMA integrity_check`，保留业务数据与安全配置，且 `sessions` 表为空。
- [x] AC6: 生成备份不会修改源数据库中的业务数据或当前 session。
- [x] AC7: 生产环境的非 HTTPS 请求被拒绝；开发环境仍可在本地 HTTP 下验证功能。
- [x] AC8: 备份失败返回明确但不泄漏内部路径或凭据的错误，并清理所有临时产物。
- [x] AC9: 自动化测试覆盖重新认证、可选 TOTP、一致性备份、会话清理、源库不变和失败清理。
- [x] AC10: README 或部署文档包含备份下载与停机恢复步骤。

## Out of Scope

- 定时备份、备份保留策略、NAS 或云存储集成。
- 应用内恢复、数据合并、跨 schema 版本降级恢复。
- 备份文件应用层加密、密码压缩包或密钥管理。
- 新数据库引擎或数据库 schema 迁移。

## Notes

- 本任务跨越 UI、API、认证、SQLite、测试和部署文档，按复杂任务处理。
- 当前开发数据库约 40 KiB；MVP 可在生成并校验后将文件读入内存响应。若未来数据库显著增大，应改为可清理的流式下载方案。

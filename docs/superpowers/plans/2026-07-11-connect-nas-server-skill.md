````markdown
---
name: connect-nas-server
description: Use when the user asks to connect to, inspect, transfer files to or from, troubleshoot, or routinely administer the Linux NAS through the nas-server SSH alias.
---

# Connect NAS Server

## Overview

Use `ssh nas-server` as the only connection entry point. Inspect first, make the smallest scoped change, and verify afterward. Never read, copy, display, or embed the private key.

## Workflow

1. Classify the request as inspection, routine mutation, or high risk.
2. Run a read-only preflight to confirm paths, permissions, service names, and available tools.
3. For routine mutations, state the exact change and back up important configuration with a timestamp.
4. Execute the smallest command that satisfies the request.
5. Verify with an independent read-only command.
6. Report changes, evidence, backup path, and limitations.

## Commands

| Task | Pattern |
| --- | --- |
| Remote command | `ssh nas-server -- '<command>'` |
| Upload | `scp -- <local-path> nas-server:<remote-path>` |
| Download | `scp -- nas-server:<remote-path> <local-path>` |
| Interactive shell | `ssh nas-server` only when interaction is necessary |

Quote local PowerShell paths and remote shell paths independently. Prefer bounded connection timeouts and non-interactive commands. Detect `systemctl`, `service`, Docker, Podman, and package managers before using them.

## Routine Management

Read system state, logs, storage, processes, services, and containers. Transfer user-requested files, create scoped paths, edit configuration after inspection and backup, and manage an explicitly named service or container when the request authorizes it. Verify every mutation.

If `sudo` needs a password, never store or inject it. Ask the user to use a secure interactive mechanism.

## Confirmation Gate

Obtain explicit command-specific confirmation immediately before:

- recursive deletion, bulk move, important overwrite, or broad glob expansion;
- reboot, shutdown, suspend, or likely disconnection;
- partition, filesystem, RAID, LVM, mount-table, or destructive repair work;
- firewall, routing, DNS, SSH, key, user, group, `sudoers`, or authentication changes;
- system-wide upgrades or dependency-removing package operations;
- pruning containers, images, volumes, snapshots, backups, or recovery data;
- any command whose resolved scope cannot be verified.

One approval does not authorize later high-risk commands.

## Host Identity And Failures

On a new or changed SSH host key, stop and compare the fingerprint with a trusted value from the user or NAS console. Never use `StrictHostKeyChecking=no`, delete a known-host entry automatically, or accept identity based only on network discovery.

Distinguish timeout, SSH config error, host-key mismatch, authentication failure, and remote permission failure. After a partial mutation, stop further changes, inspect current state, and propose recovery from the backup.

## Example

For a configuration update: inspect the target and service, create `target.bak-YYYYMMDD-HHMMSS`, upload to a temporary path, validate syntax if supported, atomically install the file, restart only the named service, then verify service status and relevant logs.

## Common Mistakes

- Repeating IP, user, port, or key path instead of using `nas-server`.
- Assuming systemd, Docker, a distribution, or a package manager exists.
- Printing broad environment or configuration output that may contain secrets.
- Treating a service restart request as permission to reboot the host.
- Claiming success without a separate verification command.
````
